00

The Big Picture

Mental Model
Core Philosophy When a user says "schedule my day", the agent doesn't start from scratch each time. It already knows who this person is — a college student vs. a working professional, a morning person vs. a night owl, someone who skips gym on Mondays. The schedule it produces is an expression of accumulated understanding, not a blank-slate answer.
Three Phases Every Time Phase 1 — Identity & Context Resolution: Who is this user? What does the agent already know? What time is it, what day is it, what does their week look like? Pull all of this from the memory layers before doing anything else.
Phase 2 — Intent Decomposition & Clarification: What exactly do they want? "Schedule my day" vs "schedule my week" vs "I have an exam Friday" all require different planning graphs. If context is missing, ask one targeted question before proceeding.
Phase 3 — Schedule Construction & Confirmation (HITL): Generate a proposed schedule using the user's profile + constraints, present it for review, accept edits, then commit to Firestore and set push notifications.

How the Agent Knows Who You Are

User Profile System
Two Sources of Identity The agent builds a user profile from two streams: explicit information (things the user directly told it during onboarding or in conversation) and inferred information (patterns the agent observes over time — you always cancel gym on Tuesdays, you study better after 9 PM, you prefer 45-min blocks). Both live in the LangGraph Store (Redis) scoped to the user's Firebase UID.

How the Agent Learns on Day 1

First-Time Onboarding
Conversational Slot Filling The agent doesn't show the user a form. It has a conversation. Using LangGraph's HITL interrupt pattern, it pauses at each missing "slot" in the user profile and asks one targeted question at a time. The LLM decides which question to ask next based on what's already known. This is called adaptive slot filling — the order and phrasing adapts to what the user has already revealed.
First Launch — Onboarding Conversation

Hey! I'm your personal AI agent. I'll handle your schedule, study reminders, and even curate movies for your weekends — but first I need to understand your life a bit. Are you a student, working professional, or both?

03

When User Says "Schedule My Day"

LangGraph Execution Flow
What Gets Triggered This single sentence kicks off a LangGraph graph execution with 7 distinct nodes. The agent doesn't just answer — it reasons, retrieves, plans, checks, proposes, waits for approval, then commits. Each step is a node. The path between them changes based on what it finds.
1
Intent Classification Node
The Root Orchestrator LLM reads the message and classifies it. "Schedule my day" → scope: single_day, intent: schedule_build. "Schedule my week" → scope: full_week. "I have an exam Friday" → intent: schedule_adjust, constraint: exam_day=Friday. This classification determines which sub-graph to execute next.
// Classified intent object passed into graph state { "intent": "schedule_build", "scope": "single_day", "target_date": "2026-06-08", // today or specified "constraints": [], "needs_clarification": false }
2
Context Hydration Node
Before any planning, the agent loads all relevant context. It pulls from three sources simultaneously: (A) the LangGraph Store (Redis) for the user profile and long-term preferences, (B) Firestore via MCP tool for today's specific timetable and any existing calendar events, (C) Gemini Nano on-device (via ADK) for current location and time context. All three run in parallel as LangGraph parallel branches.
// Parallel context fetch — all 3 fire at once await Promise.all([ store.get(["users", uid]), // Redis: user profile mcp.call("timetable_fetch", {date}), // Firestore: today's classes adk.onDevice("get_context") // Gemini Nano: time + location ])
3
Clarification Check Node (HITL — Conditional)
The agent checks if it has enough context to build a good schedule. If the user profile is complete and the date is clear → skip to planning. If something critical is missing (e.g., it's a new week and no exam dates are known) → interrupt graph, ask one targeted question, wait. LangGraph's interrupt_before pauses execution and saves state via the checkpointer. The conversation can sit paused for hours until the user responds.
// Conditional edge: missing context → ask, else plan def route_after_context(state): if state["missing_slots"]: return "ask_clarification" // → HITL interrupt node return "build_schedule" // → planning node // Example clarification: "Any exams or deadlines this week I should know about?"
4
Schedule Builder Node — The Planning Brain
This is where the actual intelligence lives. The Gemini Flash LLM receives a carefully constructed prompt containing: the user's full profile, today's fixed constraints (college classes, internship days), preferences (study block length, peak focus time), inferred patterns (which gym days they usually skip), and the current day/time. It outputs a structured JSON schedule. The schedule respects:
— Fixed blocks: college classes, internship hours (non-negotiable)
— Flexible blocks: study, gym, meals, personal time (slotted around fixed)
— Inferred adjustments: if today is Thursday and they have a late lab, study is pushed to post-dinner
— Mode awareness: weekday = structured, weekend = relaxed with movie slot
// Simplified prompt structure sent to Gemini Flash """ You are scheduling for a 3rd-year CS student at RVCE. Today: Monday June 8, 2026 FIXED (non-negotiable): - DSA lecture: 9:00 AM – 10:00 AM - Physics lecture: 11:00 AM – 12:00 PM USER PROFILE: - Wake time: 6:30 AM | Peak focus: evening - Study block preference: 90-min Pomodoro - Gym: NOT today (Monday — historically skips) - Dinner: ~8 PM | Sleep target: 11:30 PM TASK QUEUE (from Firestore): - DBMS assignment due Wednesday - Physics lab report due Friday Build a structured day schedule as JSON with time, title, type, and notes. """
5
Schedule Proposal — HITL Review (interrupt_before)
The agent never silently commits a full schedule. It presents the proposed schedule to the user in a clean visual format (the mobile app renders time blocks) and waits. The graph is paused via LangGraph checkpointer. The user can: approve (→ commit), edit specific blocks (→ update_state() → re-run planner for affected blocks), or reject and re-plan (→ restart planning node with new constraints). This is the HITL "human as judge" pattern.
// User edits a block — agent updates state and re-evaluates downstream graph.update_state(config, { "schedule_edits": [{ "block": "gym", "action": "move", "new_time": "7:00 PM" // user pushed gym 30 mins later }] }) graph.invoke(Command(resume={"approved": true}), config)
6
Commit & Notify Node
Once approved, the agent fires three MCP tools in parallel: calendar_write (saves all blocks to device calendar + Firestore), push_schedule (registers Expo push notifications 15 min before each block), and memory_update (if the user edited any blocks, those edits are stored in the LangGraph Store as behavioral signals — e.g., "user prefers gym at 7 PM not 6:30 PM").
// Three parallel MCP tool calls on approval await Promise.all([ mcp.call("calendar_write", {blocks: schedule}), mcp.call("push_schedule", {alerts: schedule.map(reminderFor)}), mcp.call("preference_store", {uid, edits: userEdits}) ])
7
Continuous Learning Node (Background)
As the day progresses, the agent monitors completion. Did the user actually wake up at 6:30? Did they skip the study block? Did they extend the gym session? These signals are processed asynchronously (not in the main graph — a separate background LangGraph cron node runs every night) and written back to the LangGraph Store. Over 2 weeks, the agent's schedule proposals get noticeably more accurate to the user's real behavior. This is episodic memory informing future planning.
// Nightly learning node — runs via cron, updates user profile def learning_node(state): completed = state["completed_blocks"] skipped = state["skipped_blocks"] # Update skip probability in Store for block in skipped: profile["reschedule_patterns"][block] += 0.1 store.put(["users", uid], "profile", profile)

Intent → Agent Routing

LangGraph Conditional Edges
How One Sentence Routes to the Right Agent Every message goes through the Intent Classification Node first. The output determines which specialist agent in the LangGraph graph handles it next. These are conditional edges in LangGraph v1 — the graph literally branches differently based on what the LLM classified.
User Says Intent Agent Routed To Memory Accessed
"Schedule my day" schedule_build · day Routine Agent → Schedule Builder Node. Pulls full profile, today's timetable, task queue. Store (profile) + Firestore (timetable) + Gemini Nano (time context)
"Schedule my week" schedule_build · week Routine Agent loops over 7 days, runs planning node per day, applies cross-day constraints (exam proximity, assignment deadlines). Store + Firestore weekly timetable + task queue
"I have an exam Friday" schedule_adjust · constraint_add Academic Agent. Writes exam to task queue (priority: critical). Triggers reschedule of Fri study blocks. Suggests adding a revision session Thu night. Firestore (writes exam event) + Store (updates priority weights)
"Move gym to tomorrow" schedule_edit · block_move Routine Agent. Updates today's and tomorrow's schedule. Checks for conflicts. Updates push notifications via MCP push_schedule tool. Checkpointer (current schedule state) + MCP (calendar + push)
"What should I watch tonight?" entertainment_rec · tonight Entertainment Agent. Checks if it's weekend (TMDB API full search) or weekday (shorter runtime filter). Fetches 3 curated options based on genre history. Store (movie preferences, watch history) + TMDB MCP tool
"Remind me about my DBMS assignment" reminder_set · academic Academic Agent. Writes reminder to task queue with priority + deadline. Schedules push notification via MCP. Updates study blocks to include DBMS time. Firestore (task queue) + MCP (push_schedule)
"I'm near the gym" location_trigger · gym_proximity Location Agent (on-device, Gemini Nano). Geofence match. Checks if it's a gym day and it's within the gym window. Sends push: "You're near the gym — it's your gym day, want to go now?" Store (gym_days) + ADK on-device only — no cloud round-trip
"I'm exhausted, skip study today" schedule_override · user_state Routine Agent + HITL. Acknowledges, removes study blocks, asks: "Should I reschedule DBMS assignment to tomorrow?" Stores exhaustion event as episodic memory signal. Checkpointer (live schedule) + Store (writes fatigue pattern signal)

How Memory Makes It Personal Over Time

3 Memory Types in Play
⚡
Semantic Memory — "Know Facts About You"
LangGraph Store · Redis · cross-session · always available
Stores facts about the user that the agent explicitly learns. Name, role, schedule preferences, college timetable, gym days. This is the backbone of personalization — the agent queries this before every scheduling task to know who it's talking to and what their constraints are.
Example fact stored: {"key": "gym_days", "value": ["Tue", "Thu", "Sat"], "source": "explicit"}
🔁
Episodic Memory — "Remember What Happened"
LangGraph Store · Redis · nightly background write · cross-session
Stores past events and their outcomes. "User skipped gym Monday 4 out of 5 Mondays." "User extended study block by 30 min on 3 occasions." The nightly learning cron node processes completion signals and updates behavioral probability weights. After 2 weeks, schedule proposals are noticeably adapted to actual behavior.
Example: {"event": "gym_skip", "day": "Monday", "count": 4, "skip_probability": 0.8}
📋
Procedural Memory — "How to Plan For You"
System prompt optimization · evolves with feedback
The agent's planning instructions aren't static. After the user edits a schedule (pushes gym later, reorders study blocks), those edit patterns get compressed into updated planning instructions. Over time the system prompt itself evolves to reflect the user's planning style — it knows to put gym at 7 PM not 6:30, and to never schedule deep work right after a long lab.
Instruction evolved: "This user prefers gym sessions at 7 PM. Never schedule study blocks within 30 min of meals."

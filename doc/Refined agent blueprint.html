Protocol Architecture: MCP + A2A

2026 Standards
MCP
Model Context Protocol · Agent ↔ Tool
MCP is the nervous system — it connects your agents to tools and data sources. Every external capability your agent needs (calendar, TMDB, notifications, Firestore) is exposed as an MCP tool with a JSON Schema definition. The agent calls tools without knowing their implementation.
calendar_read timetable_fetch tmdb_search push_schedule geofence_set preference_store movie_history
A2A
Agent2Agent Protocol · Agent ↔ Agent
A2A is the communication network — it lets agents built on different frameworks discover and delegate to each other. Your Root Orchestrator can delegate the movie recommendation task to the Entertainment Agent via A2A's Agent Card system, passing structured Task objects with lifecycle states.
Agent Cards (discovery) Task lifecycle SSE streaming JSON-RPC / HTTP Cross-framework
How They Work Together Think of it this way: MCP gives each agent its tools (a screwdriver, a hammer). A2A lets agents hire each other (one contractor delegating to a specialist). Both run simultaneously in production — LangGraph handles MCP tool calls natively; ADK Python 2.0 adds A2A Agent Cards so your agents can be discovered and orchestrated by external systems, including Android's system Gemini.

Memory Architecture

LangGraph v1 Dual-Layer
Critical Design Note LangGraph v1 has two separate memory mechanisms — most apps only implement one and break in production. A Checkpointer handles session continuity (within a thread_id). A Store handles cross-session continuity (across thread_ids). Your user's gym schedule should survive a new chat session — that requires the Store, not the Checkpointer.
Layer 1
In-Context Window
The current conversation messages, tool call history, and current routine state. Managed automatically by LangGraph's MessagesState. Lives only for the duration of one graph execution.
scope: single graph run
backend: MessagesState
lifetime: ephemeral
Layer 2
Session Memory (Checkpointer)
Persists the full conversation state within one thread_id. Allows a user to pick up a conversation hours later. AsyncPostgresSaver is the production choice — SQLite bottlenecks under load, InMemorySaver evaporates on restart.
scope: per thread_id
backend: AsyncPostgresSaver
lifetime: session duration
Layer 3
Long-Term Memory (Store)
Cross-session facts, preferences, and habits. When a user returns next week with a new thread_id, the agent still knows their gym schedule, subject preferences, and movie taste. Redis Store for <1ms retrieval with Firestore as backup sync.
scope: cross-session (user_id)
backend: Redis Store + Firestore
lifetime: permanent

Hybrid Inference Model

ADK for Android
The Architectural Shift Google ADK for Android (launched May 2026) enables a hybrid orchestration pattern: the cloud model acts as root orchestrator while on-device Gemini Nano handles sub-agents for privacy-sensitive and low-latency tasks. This means your app can send routine reminders even with no internet, and never sends the user's location or calendar to the cloud.
On-Device (Gemini Nano)
  • Routine alarms — wake-up, meal, study reminders
  • Intent classification — is this a schedule or movie query?
  • Location processing — geofence eval, gym proximity check
  • Calendar read/write — local device calendar access
  • Offline fallback — full routine mode without cloud
  • Privacy-safe — sensitive data never leaves device
ADK Hybrid Bridge
Cloud (Gemini 2.0 Flash)
  • Root orchestration — complex multi-step reasoning
  • TMDB curation — personalized movie recommendations
  • Long-term memory writes — learning user preferences over time
  • Academic planning — timetable optimization across weeks
  • Natural language chat — freeform conversation interface
  • A2A agent delegation — publish Agent Card for external use
